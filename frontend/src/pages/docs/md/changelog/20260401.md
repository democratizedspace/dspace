DSPACE v3 is a major content and platform release focused on three goals:

1. **Much more to do** (quests, processes, and items)
2. **Better creator workflows** (build and submit [custom content](/docs/custom-content) in-game)
3. **Safer, local-first player data** (migration, backups, and cloud sync)

If you are new to DSPACE, start with [About](/docs/about).

## v3 at a glance (vs v2.1)

The current v3 release snapshot includes:

- **248 quests** across **19 quest trees**
- **311 processes**
- **507 items**

Compared with v2.1 (current `main`):

| Category  | v2.1 | v3 | New in v3 |
| --------- | ---: | -: | --------: |
| Quests    | 22   | 248 | **+226** |
| Processes | 36   | 311 | **+275** |
| Items     | 90   | 507 | **+417** |

That means v3 delivers roughly **11.3x more quests** than v2.1, exceeding the original 10x target.

## Custom content (headline v3 feature)

v3 ships end-to-end creator tooling for player-generated content:

- [Custom quest creation](/quests/create), including validation and preview
- [Custom item creation](/inventory/create) (with `/items/create` preserved as a legacy alias)
- [Custom process creation](/processes/create), including requires/consumes/creates modeling
- [Bundle submission workflow](/bundles/submit) for grouped quest/item/process contributions

Like custom quests, your creations are stored locally by default. When you're ready to contribute,
package everything into a bundle following the [Custom Content Bundles](/docs/custom-content-bundles)
format and submit it through the in-game interface. This keeps related content synchronized and
makes reviews more efficient.

Reference docs:

- [Quest Submission Guide](/docs/quest-submission)
- [Custom Content Bundles](/docs/custom-content-bundles)

## Local-first saves, migration, and backups

v3 moves canonical save storage to IndexedDB and keeps migration from legacy v2 saves built in.

- On first launch, v3 detects legacy `localStorage` save keys and migrates data to IndexedDB.
- Save and content backup workflows are available in-game.
- Cloud backup and restore is available through GitHub Gist integration.

## AI chat in v3

v3 launches with **OpenAI-only chat** in production.

- NPC personas (such as Sydney, Nova, and Hydro) are available in chat.
- token.place integration is coming in v3.1, bringing free LLM usage instead of the metered
  usage that currently requires an OpenAI API key.

See:

- [v3 Release State](/docs/v3-release-state)
- [token.place rollout notes](/docs/token-place)

### dChat knowledge upgrade

dChat now ships with an in-game knowledge base that summarizes canonical quests, items, processes,
and the player's local inventory snapshot. This gives the assistant immediate context to answer
gameplay questions without waiting for future updates or external documentation lookups.

## Self-hosted

DSPACE 2.1 was migrated from Netlify to a local Mac Mini M4. DSPACE v3 now runs on-premises on a
k3s cluster across 9 Raspberry Pis.

## Switching to MIT License

While 0BSD sounds like a nice license, it is in some cases not on the list of accepted OSS licenses
on the Internet. Therefore, I'm switching from 0BSD to MIT, which is still very permissive but much
better supported.

## Looking Forward

These changes reflect a commitment to keeping DSPACE focused on its core mission: democratizing
space exploration through education, simulation, and real-world impact. By removing unnecessary
complexity, the game is more accessible while strengthening the connection between virtual
achievements and actual space development.

v3 is the biggest DSPACE update yet, and it lays the groundwork for faster content releases moving
forward.

## June 1, 2026 — DSPACE v3.0.1 (patch addendum)

`v3.0.1` is a patch update to the April 1 v3 release, focused on launch-readiness
hardening across the audited delta
`3ec45a5517a35c96767f6b946c01104e6ec88f93..2d7f93daa4869ae223825cbd00a37bc83ac5703e`.

### Patch notes

- **`/quests` performance + correctness:** improved list-path time-to-interactive, tightened built-in/custom coexistence behavior, fixed locked-quest status visibility regressions, and removed dead-end gating in preflight-check quest flow.
  - Cold run: list visible 30.3 ms → 0.3 ms (~101x faster); full reconciliation 72.6 ms → 0.7 ms (~104x faster).
  - 4x CPU-throttled run: list visible 115.5 ms → 1.9 ms (~61x faster); full reconciliation 262.6 ms → 6.2 ms (~42x faster).
- **Gameplay/data integrity hardening:** normalized stored item counts, rejected unknown item containers, aligned geothermal reward docs with quest data, and sanitized compact item/process preview image handling.
- **`/quests` security:** sanitized custom quest tile routes to fall back to internal `/quests/{id}` links when values are unsafe, including protocol-relative prefixes like `//` and slash-backslash variants like `/\`.
- **`/processes` reliability:** switched list rows to summary-first rendering while retaining runtime controls on `/processes/:processId`, and hardened buy-required metadata-loading behavior plus regression coverage.
- **`/docs` and chat retrieval:** deferred `/docs` full-text corpus fetch until first keyword search (with immediate browse/`has:` access), plus hardened docs-index/chat smoke assertions.
- **Migration + remote-smoke harness hardening:** fixed remote legacy migration real-v2 coverage and strengthened remote smoke flow reliability for custom-item mutations and live-chat checks.
- **Release governance/runbook updates:** tightened v3.0.1 QA checklists/evidence gates, added canonical release-history tracking, and clarified rollback/promotion runbook links and compare-based PR references.

## DSPACE v3.1 — main branch updates since v3

`v3.1` is the current main-branch update track after the April 1 v3 baseline. The summary below
covers user-visible and launch-relevant work audited across
`3ec45a5517a35c96767f6b946c01104e6ec88f93..2410f278a340d9ea3e164f203c540fdd4174d126`.
It does **not** claim staging approval, production promotion, or a final release tag.

### Faster, clearer quest browsing

- **`/quests` reaches useful content sooner.** The built-in quest list now renders the actionable
  grid first, then reconciles slower custom quest data afterward so local custom content does not
  block the initial list experience.
- **Built-in and custom quests coexist more reliably.** Delayed custom quest loading keeps custom
  cards from displacing built-ins, avoids duplicate/corrupted IDs, and preserves navigation from
  list to detail through refreshes and mixed built-in/custom sessions.
- **Quest status labels are less confusing.** Locked or unavailable quests no longer flash an
  optimistic start/continue state, completed built-ins remain in the completed section, and
  completed custom quests are grouped with completed quest history instead of being treated as a
  separate odd case.
- **Quest detail loading is steadier.** Quest chat readiness, gate refreshes, and route fallback
  behavior were tightened so players see shorter loading copy and fewer transient gate/status
  flashes.

### Process lists, details, and buy flows

- **`/processes` is summary-first.** The list route now shows lightweight rows with the title,
  duration, and requirement/consume/create counts, while keeping runtime controls such as start,
  pause, resume, cancel, collect, and buy actions on the detail route.
- **Process previews avoid placeholder noise.** Metadata-dependent previews no longer expose raw
  unresolved item IDs while item data is still loading, and built-in processes remain visible before
  custom processes merge in.
- **Buying process inputs is more dependable.** The `Buy required items` flow now accounts for both
  required and consumed inputs, supports consume-only processes, keeps purchasable fallbacks working,
  and behaves consistently for embedded process views on item pages.
- **Custom process coexistence was hardened.** Custom process rows merge after the initial list load,
  remain identifiable as custom content, and keep detail links working alongside active built-in
  process progress.

### Docs search and dChat grounding

- **`/docs` avoids a heavy first payload.** Browse mode and metadata filters now work without
  downloading the full text corpus up front.
- **Search behavior stays expressive.** `has:` queries such as `has:link` and `has:image` continue to
  work before the full corpus is fetched, while the first keyword search loads
  `/docs/full-text-corpus.json` on demand.
- **Corpus failures are recoverable.** If full-text loading fails, docs search degrades gracefully and
  can retry instead of leaving the page stuck in an empty or broken state.
- **dChat remains docs-backed.** Chat regression coverage was updated around the docs corpus changes
  so launch-check prompts still return relevant, non-empty answers grounded in the expected DSPACE
  documentation topics.

### Custom content, saves, and account reliability

- **Custom-content imports are safer.** Imported custom content and shop links are sanitized more
  aggressively, including unsafe quest tile routes, compact item/process preview images, unknown
  item containers, and malformed link targets.
- **Inventory counts and containers are stricter.** Stored item counts are normalized, duplicate
  lookup paths were reduced, and unknown containers are rejected instead of silently corrupting
  inventory state.
- **Cloud Sync is more tolerant of account timing.** Authentication readiness, backup refreshes, and
  sync-success flows were hardened so players are less likely to see stale or race-prone cloud-sync
  UI after sign-in, logout, or backup updates.
- **Settings are easier to scan.** The settings page now uses a responsive grid and unified card
  styling so account, backup, and data-management controls are clearer across screen sizes.

### Release safety and operator-facing polish

- **Release records are easier to audit.** v3.0.1 candidate metadata, release history, rollback
  references, and staging/prod evidence gates were tightened so launch reviewers can distinguish
  verified evidence from pending deployment work.
- **Smoke and regression harnesses cover the launch-critical routes.** Remote smoke, legacy save
  migration, `/quests`, `/processes`, `/docs`, `/chat`, custom content, and authentication flows now
  have more targeted checks for the route behaviors summarized above.
- **Build/test output is less noisy.** Known benign Playwright, service-worker, Browserslist, npm,
  and Node warning noise is filtered or documented so release failures are easier to spot without
  hiding real route regressions.
