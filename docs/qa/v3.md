# DSPACE v3 QA Checklist (Release Candidate → Production)

> Goal: ship v3 with **zero broken routes**, **working core gameplay (quests/processes/items)**,
> **working custom content (custom quests/processes/items)**, **docs + changelog accurate**, and
> **AI chat stable** (provider(s) verified, graceful failure, no regressions).
>
> Suggested process: do Automation → Route/Smoke → Core Systems → Custom Content → Docs/Changelog → Prod smoke.

---

## Quick command index

- `npm run qa:smoke` — type-checks, link-checks, validates items, and runs CI-parity tests without
  Playwright E2E
- `npm run test:pr` — full pre-PR suite (root tests, hardening metadata, lint/format check, grouped
  Playwright E2E)
- `npm run link-check` — validates internal docs routes and external links
- `npm run itemValidation` — runs schema validation for built-in items, processes, and quests
- `python -m scripts.image_issues find-image-issues` — required release gate for missing/duplicate
  images (see [Developer Guide](../../DEVELOPER_GUIDE.md#image-analysis-cli))

Related docs: [QA index](./README.md) and [reusable template](./template.md).

## 4) Core gameplay QA: Quests (built-in content)

### 4.1 Baseline quest flow
- [ ] Start a fresh profile and complete the onboarding/welcome flow (no dead ends)
- [ ] Quest list renders, filters work, categories behave
- [ ] Quest detail view renders:
  - [ ] Dialogue nodes visible
  - [ ] Options selectable
  - [ ] Locked options clearly explained
  - [ ] Images render (no broken/missing/duplicated visuals)
- [ ] Completing a quest updates:
  - [ ] Quest state (completed)
  - [ ] Inventory changes (if rewards)
  - [ ] Processes unlocked/started (if applicable)

### 4.2 Dependency graph quality audit (human judgment)
- [ ] Review each quest category for “sense-making progression”
  - [ ] Hydroponics
  - [ ] Electronics & Robotics
  - [ ] Astronomy/Observation
  - [ ] Energy/Power
  - [ ] Aquaria
  - [ ] Programming/Devops/Sysadmin
  - [ ] Fabrication/3D Printing
  - [ ] Chemistry/First Aid/Safety
  - [ ] Rocketry/Flight
  - [ ] Woodworking/Construction
  - [ ] Completionist/Welcome/Other meta categories
- [ ] For **each category**, pick at least **one representative chain** and verify:
  - [ ] No awkward gating (requires obscure items without explanation)
  - [ ] Dependencies feel “teachable” (you learn what you need before you need it)
  - [ ] Rewards match the effort and unlock something meaningful

### 4.3 Quest rewards sanity
- [ ] Rewards are not wildly inflationary (economy doesn’t break)
- [ ] Rewards are not stingy to the point of stalling progress
- [ ] Item reward images are correct and non-duplicative where it matters
- [ ] Process-launch rewards actually start/enable the intended process
- [ ] Any “special” rewards (titles/achievements) trigger reliably

### 4.4 Quest copy/narrative QA
- [ ] No obvious placeholder text / unfinished sections
- [ ] Tone consistency within a chain
- [ ] Option text clearly signals consequences
- [ ] Typos/grammar pass (or tracked issues created)

## 7) Custom content QA (the v3 headline feature)

> The golden path here is: create custom item → create custom process using it →
> create custom quest that rewards it / triggers it → preview → save → export → reset → import → verify.

### 7.1 Custom Items
- [ ] Open `/inventory/manage`
- [ ] Create a custom item:
  - [ ] Validates required fields
  - [ ] Image upload works (no corruption, reasonable size)
  - [ ] Preview renders accurately
- [ ] Edit custom item and confirm changes persist
- [ ] Delete custom item and confirm removal everywhere it’s referenced (or blocked with a clear error)

### 7.2 Custom Processes
- [ ] Open `/processes/manage`
- [ ] Create a custom process:
  - [ ] Duration parsing accepts valid formats and normalizes
  - [ ] Requires/Consumes/Creates can reference both built-in + custom items
  - [ ] Validation errors are clear and actionable
- [ ] Start/cancel/collect with a custom process:
  - [ ] Inventory deltas correct
  - [ ] No stuck states after reload

### 7.3 Custom Quests (editor + simulation)
- [ ] Open `/quests/manage`
- [ ] Create a custom quest using the in-game editor:
  - [ ] Nodes/options can be added/edited/deleted
  - [ ] Images can be attached and display correctly
  - [ ] Schema validation blocks invalid saves
  - [ ] Preview mode matches the live quest experience
  - [ ] Simulation tests run and catch obvious logic errors
- [ ] Create a quest dependency on:
  - [ ] A built-in quest
  - [ ] Another custom quest
- [ ] Create quest rewards:
  - [ ] Reward a custom item
  - [ ] Trigger a custom process
- [ ] Confirm custom quests appear in `/quests` list and behave like built-ins

### 7.4 Quest PR submission (GitHub workflow)
- [ ] Open `/quests/submit`
- [ ] With a GitHub token, create a PR successfully
- [ ] Failure modes are safe + clear:
  - [ ] Missing token → blocked with instructions
  - [ ] Bad scopes → clear error
  - [ ] Network/API failure → clear error, no data loss
- [ ] Confirm secrets are not logged or embedded in exported data

## 9) AI chat QA (OpenAI + token.place plan)

> Principle: chat must either work, or fail *politely* with clear errors and no broken UI.

### 9.1 `/chat` UX + stability
- [ ] Chat page loads and persona picker works
- [ ] Sending a message:
  - [ ] Shows loading state
  - [ ] Returns a reply
  - [ ] Handles long replies (scroll, wrap, etc.)
- [ ] Error handling:
  - [ ] Network failure shows user-facing error
  - [ ] Rate limit / provider error shows user-facing error
  - [ ] Abort/cancel (if supported) stops cleanly

### 9.2 Provider verification (choose what v3 officially supports)
- [ ] Confirm what provider is “official” for v3:
  - [ ] OpenAI-only
  - [ ] token.place-only
  - [ ] Both with a toggle/fallback
- [ ] Confirm docs match reality (no “docs say token.place but app uses OpenAI” drift)

### 9.3 token.place decision (recommended: v3.1 unless already fully deployed)
- [ ] If token.place is enabled in v3:
  - [ ] Endpoint configuration works (`VITE_TOKEN_PLACE_URL` and/or in-game setting)
  - [ ] Default endpoint works in staging/prod
  - [ ] Failover behavior is defined (fallback or hard error)
- [ ] If token.place is deferred to v3.1:
  - [ ] v3 docs do not over-promise token.place availability
  - [ ] v3 UI hides token.place-only options or labels as experimental
