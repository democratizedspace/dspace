# DSPACE v3 QA Checklist (Release Candidate → Production)

> Goal: ship v3 with **zero broken routes**, **working core gameplay (quests/processes/items)**,
> **working custom content (custom quests/processes/items)**, **docs + changelog accurate**, and
> **AI chat stable** (provider(s) verified, graceful failure, no regressions).
>
> Suggested process: do Automation → Route/Smoke → Core Systems → Custom Content → Docs/Changelog → Prod smoke.

---

## Quick command index

- `npm run qa:smoke` — type-checks, link-checks, validates items, and runs CI-parity tests without
  Playwright E2E
- `npm run test:pr` — full pre-PR suite (root tests, hardening metadata, lint/format check, grouped
  Playwright E2E)
- `npm run link-check` — validates internal docs routes and external links
- `npm run itemValidation` — runs schema validation for built-in items, processes, and quests
- `python -m scripts.image_issues find-image-issues` — required release gate for missing/duplicate
  images (see [Developer Guide](../../DEVELOPER_GUIDE.md#image-analysis-cli))

Related docs: [QA index](./README.md) and [reusable template](./template.md).

### 3.2.2 Pre-launch: Legacy save migration safety (v1/v2 → v3)

How to use this checklist: run it once on staging before launch, and re-run any item after
changing migration/seed logic. Record outcomes inline as you test.

#### A) Safety invariants (must-haves)
- [ ] Migration writes v3 data first, then handles legacy cleanup per current behavior:
  - [ ] v1 cookies are expired as part of the v1 migration action.
  - [ ] v2 auto-migration removes `gameState`/`gameStateBackup` from localStorage after the v3
        write succeeds.
  - [ ] v2 manual migration leaves legacy localStorage intact unless “Discard legacy v2 data” or
        “Clear seeded legacy saves” is explicitly invoked.
- [ ] A recovery path exists for users (must verify / implement before launch):
  - [ ] Export v3 save to a downloadable JSON file (or documented manual export steps if export UI
        isn’t implemented yet).
  - [ ] Restore v3 save from JSON (or documented manual restore steps if restore UI isn’t
        implemented yet).
  - [ ] Documentation states where v3 data lives (IndexedDB) and how to clear/reset safely for QA.
- [ ] Migration is idempotent / runs once per legacy source:
  - [ ] Legacy data deletion is the primary protection against re-running migration.
  - [ ] If explicit “already migrated” markers are required, mark as “must implement before
        launch.”
  - [ ] Documentation explains merge vs replace semantics and what happens if legacy data is
        re-seeded after a migration (must verify / implement before launch).
- [ ] Migration is atomic enough to avoid partial state:
  - [ ] If migration fails midway, confirm the current behavior (logs/errors, no “incomplete”
        marker) and decide whether an explicit retry UX is required before launch.

#### B) Pre-launch manual QA matrix (v1 → v3)
- [ ] Clean v3 environment (no cookies/localStorage/IndexedDB); confirm no legacy banner/cards.
- [ ] Seed minimal v1 legacy state; confirm detection UI (banner + /settings card) is correct.
- [ ] Seed maximal v1 legacy state; confirm detection UI is correct.
- [ ] Merge v1 into an existing v3 save (non-empty v3 inventory/wallet): verify additive behavior is
      correct.
- [ ] Replace v1 into an existing v3 save: verify overwrite behavior is correct.
- [ ] Verify post-migration:
  - [ ] v3 IndexedDB contains expected items + currency.
  - [ ] Legacy artifacts match current behavior (v1 cookies expire on migration; v2 auto-migration
        removes localStorage after successful write; manual v2 migration leaves data until
        explicitly discarded).
  - [ ] Banner disappears only when appropriate (define expected behavior clearly).
- [ ] v1 cookies contain only item counts; quest progress and process timers did not exist in v1
      and are therefore not part of v1 → v3 migration.

#### C) Pre-launch manual QA matrix (v2 → v3)
- [ ] Seed/construct a minimal v2 `localStorage.gameState` JSON blob with:
  - [ ] inventory containing a float count and an int count.
  - [ ] at least one quest state.
  - [ ] at least one in-progress process (startedAt + duration).
  - [ ] Confirm v3 detects it as a v2 legacy save.
- [ ] Test migration from a real v2.1 save captured from a local run (paste the JSON into
      localStorage):
  - [ ] Detection.
  - [ ] Merge.
  - [ ] Replace.
- [ ] Decide and document the desired v2 in-progress process migration behavior (current behavior
      copies `processes` and preserves timers). Keep copying `processes` as-is; document timer
      semantics clearly in QA notes, and additionally implement and verify a migration payout for
      v2 processes that started but were not finished at v3 launch (data stored in localStorage) by
      granting the user the items from each process `createItems` list so players do not lose the
      value of consumed inputs.
- [ ] If v2 uses a backup key, confirm behavior when `gameState` is missing but
      `gameStateBackup` exists:
  - [ ] Migration reads `gameState` only (no fallback) unless a fallback is implemented before
        launch.
- [ ] Verify post-migration v2 cleanup behavior:
  - [ ] Auto-migration deletes legacy localStorage after successful v3 write.
  - [ ] Manual migration does not delete unless explicitly invoked.

#### D) Failure-mode / fuzz QA (must do at least once)
- [ ] Corrupt v2 `gameState` JSON (invalid JSON): v3 logs an error, does not crash, does not wipe
      anything; decide if clearer UX is required before launch.
- [ ] v2 `gameState` missing expected keys (inventory/quests/processes absent): migration uses
      defaults safely.
- [ ] v1 cookie values:
  - [ ] URL-encoded values.
  - [ ] empty strings.
  - [ ] NaN / non-numeric.
  - [ ] negative counts.
  - [ ] Confirm parsing is tolerant and does not miscredit or crash.
- [ ] Extremely large counts / many items: migration still completes and UI remains responsive.
- [ ] Storage quota edge: simulate near-full localStorage in browser; confirm v3 migration to
      IndexedDB still succeeds and UX is clear if writes fail.

#### E) Verification commands / manual inspection steps
- [ ] Inspect v1 cookies in DevTools (Application → Cookies → site origin).
- [ ] Inspect v2 localStorage `gameState` in DevTools (Application → Local Storage → site origin).
- [ ] Inspect v3 IndexedDB in DevTools (Application → IndexedDB): record the IndexedDB DB name,
      object store names, and key fields in QA notes.
- [ ] Factory reset for QA: clear cookies, localStorage, and IndexedDB for the site origin.

#### F) Post-launch validation (Lead QA)
- [ ] Validate migration on at least 2 distinct browsers/profiles with real v2.1 `gameState` (one
      “light” save, one “busy” save).
- [ ] Validate on at least 1 mobile browser (if DSPACE is used on mobile).
- [ ] Verify no double-crediting across refresh/revisit (note: re-seeding legacy data will
      re-trigger migration unless an explicit marker is implemented).
- [ ] Verify a separate explicit legacy cleanup action (label may vary; must verify / implement
      before launch) is safe and reversible.

#### G) Minimal observability expectation (docs-only)
- [ ] Before launch, confirm there is at least one way to detect migration failures quickly (e.g.,
      QA console error review, or a logged event if available).
