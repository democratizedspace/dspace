## 2) token.place API v1 relay E2EE contract

- [ ] DSPACE uses the relay-blind API v1 E2EE route sequence on the configured token.place origin:
  - [ ] `GET /api/v1/relay/servers/next`.
  - [ ] `POST /api/v1/relay/requests`.
  - [ ] `POST /api/v1/relay/responses/retrieve`.
  - [ ] Optional timeout cleanup uses `POST /api/v1/relay/requests/cancel` only if supported.
- [ ] Browser evidence shows DSPACE generates or loads a client keypair, fetches a compute node
      public key, encrypts the API v1 chat request envelope, polls for the encrypted response, and
      decrypts client-side.
- [ ] The `/api/v1/relay/requests` payload is ciphertext-only plus safe routing metadata such as
      `request_id`, `client_public_key`, selected server reference, nonce, and algorithm.
- [ ] token.place relay-owned state/logs do not receive plaintext DSPACE prompt messages,
      `PlayerState`, docs grounding, raw inventory, save data, OpenAI keys, token.place
      credentials, or other secrets.
- [ ] API v1 is treated as non-streaming.
  - [ ] Encrypted request options omit `stream` or set it to `false`.
  - [ ] Encrypted request options never send `stream: true`.
  - [ ] UI copy and spinners tolerate waiting until the full completion is ready.
- [ ] No token.place auth or API key is requested, stored, or sent.
- [ ] token.place request headers contain no `Authorization` header.
- [ ] Decrypted response validation checks protocol, version, request id, and client public key
      before assistant text is read from `api_v1_response.choices[0].message.content`.
- [ ] `usage` counters are displayed, logged, or preserved only when they arrive inside the
      decrypted token.place API v1 response.
- [ ] Optional metadata is safe and minimal.
  - [ ] Relay-visible metadata contains no secrets, prompt text, raw save data, or player inventory
        details.
  - [ ] Encrypted request options contain only safe non-secret correlation metadata when needed.
  - [ ] Response metadata is treated as optional and only trusted after decryption and envelope
        validation.
- [ ] No plaintext default DSPACE `/chat` request to `POST /api/v1/chat/completions`.
- [ ] No token.place API v2, SSE, streaming, token.place npm package, token.place auth/API key,
      `Authorization` header, legacy `/api/chat`, legacy `/chat`, `/next_server`, `/sink`,
      `/source`, `/faucet`, or `/retrieve` path is used.

## 4) Fresh-user happy path

- [ ] Clear browser storage for the target origin (IndexedDB, localStorage, Cache Storage if
      relevant, and service worker state if needed).
- [ ] Visit `/chat` as a fresh user.
- [ ] Confirm the active Chat panel uses token.place by default.
  - [ ] UI marker is visible, or DOM/test marker such as `data-provider="token-place"` is present.
- [ ] Select an NPC/persona if the UI does not already default to one.
- [ ] Send a simple game-related message.
- [ ] Verify the assistant response renders in Chat.
- [ ] Verify sources, usage, or debug details render only when available.
- [ ] Verify there is no API key prompt for the default token.place path.
- [ ] Verify there is no token.place credential prompt anywhere in the flow.

## 5) Settings provider flow

- [ ] Visit `/settings` with fresh storage.
- [ ] Confirm Chat provider defaults to token.place.
- [ ] Switch Chat provider to OpenAI.
- [ ] In local or non-production QA only, save a fake/test OpenAI key and verify the UI accepts it
      without sending it to token.place.
- [ ] Return to `/chat` and confirm OpenAI is selected.
- [ ] Clear the OpenAI key from Settings.
- [ ] Confirm OpenAI-selected Chat without a key shows helpful guidance instead of making an
      empty-key OpenAI request.
- [ ] Switch back to token.place.
- [ ] Reload `/settings` and `/chat`; confirm token.place selection persists.
- [ ] Reload after choosing OpenAI; confirm OpenAI selection persists when intentionally saved.

## 14) Rollback and mitigation plan

- [ ] Individual users can switch to OpenAI in `/settings` if they have their own OpenAI key.
- [ ] Deployment operators can temporarily point token.place URL/model to a known-good
      endpoint/model using `DSPACE_TOKEN_PLACE_URL` and `DSPACE_TOKEN_PLACE_CHAT_MODEL`; Vite variables remain fallback-only.
- [ ] Operators can publish an outage/support note that explains token.place Chat is degraded and
      OpenAI remains optional for users with a key.
- [ ] Do not make OpenAI the default again without an explicit product decision.
- [ ] Do not add token.place credentials, plaintext `/api/v1/chat/completions`, API v2, streaming,
      the token.place npm package, or deprecated legacy relay endpoints as emergency workarounds.
