## 2) token.place API v1 contract

- [ ] DSPACE calls `POST /api/v1/chat/completions` on the configured token.place origin.
- [ ] API v1 is treated as non-streaming.
    - [ ] Request JSON omits `stream` or sends `stream: false`.
    - [ ] Request JSON never sends `stream: true`.
    - [ ] UI copy and spinners tolerate waiting until the full completion is ready.
- [ ] No token.place auth or API key is requested, stored, or sent.
- [ ] token.place request headers contain no `Authorization` header.
- [ ] Response parsing reads assistant text from `choices[0].message.content`.
- [ ] `usage` counters are displayed, logged, or preserved only when token.place returns them.
- [ ] Optional `metadata` is safe and minimal.
    - [ ] Request metadata contains no secrets.
    - [ ] Request metadata contains no raw save data.
    - [ ] Request metadata contains no player inventory details.
    - [ ] Response metadata is treated as optional and only expected when non-empty request metadata
          is echoed.
- [ ] The v3.1 integration uses direct HTTPS API v1 requests.
- [ ] No token.place API v2, SSE, streaming, legacy `/api/chat`, legacy `/chat`, relay, sink,
      source, faucet, retrieve, or `next_server` path is used.

## 4) Fresh-user happy path

- [ ] Clear browser storage for the target origin (IndexedDB, localStorage, Cache Storage if
      relevant, and service worker state if needed).
- [ ] Visit `/chat` as a fresh user.
- [ ] Confirm the active Chat panel uses token.place by default.
    - [ ] UI marker is visible, or DOM/test marker such as `data-provider="token-place"` is present.
- [ ] Select an NPC/persona if the UI does not already default to one.
- [ ] Send a simple game-related message.
- [ ] Verify the assistant response renders in Chat.
- [ ] Verify sources, usage, or debug details render only when available.
- [ ] Verify there is no API key prompt for the default token.place path.
- [ ] Verify there is no token.place credential prompt anywhere in the flow.

## 5) Settings provider flow

- [ ] Visit `/settings` with fresh storage.
- [ ] Confirm Chat provider defaults to token.place.
- [ ] Switch Chat provider to OpenAI.
- [ ] In local or non-production QA only, save a fake/test OpenAI key and verify the UI accepts it
      without sending it to token.place.
- [ ] Return to `/chat` and confirm OpenAI is selected.
- [ ] Clear the OpenAI key from Settings.
- [ ] Confirm OpenAI-selected Chat without a key shows helpful guidance instead of making an
      empty-key OpenAI request.
- [ ] Switch back to token.place.
- [ ] Reload `/settings` and `/chat`; confirm token.place selection persists.
- [ ] Reload after choosing OpenAI; confirm OpenAI selection persists when intentionally saved.

## 6) Persistence and backup behavior

- [ ] IndexedDB primary path persists `settings.chatProvider` across reloads.
- [ ] Existing client-side OpenAI key storage remains local and backward-compatible.
- [ ] localStorage fallback sanity check passes in a browser/profile where IndexedDB is unavailable
      or intentionally blocked.
- [ ] Exported save/backup data includes provider preference only when that is part of normal game
      state export behavior.
- [ ] Exported save/backup data does not include token.place credentials.
- [ ] Import/restore preserves provider preference where relevant and does not invent token.place
      credentials.

## 7) Privacy and security

- [ ] token.place request has no `Authorization` header.
- [ ] token.place request sends no OpenAI key.
- [ ] token.place request sends no token.place credentials.
- [ ] token.place metadata contains no secrets.
- [ ] token.place metadata contains no raw save data.
- [ ] token.place metadata contains no player inventory details.
- [ ] OpenAI key stays local in the existing client storage path.
- [ ] No token.place credentials are stored in IndexedDB, localStorage, exported saves, logs, or
      debug panels.
- [ ] Prompt debug panel redacts or omits secrets.

- [ ] Docs RAG/context source behavior still adds relevant documentation context.
- [ ] Prompt debug panel shows the actual selected-provider payload.
- [ ] Save snapshot hint still appears where expected.

- [ ] No prompt, debug payload, doc copy, or UI state contains stale “token.place deferred”,

## 9) Error handling

For each case, verify the UI message is clear, no secrets are exposed, and the user can retry or
change provider when appropriate.

- [ ] token.place network failure.
- [ ] token.place HTTP 400 content policy response.
- [ ] token.place HTTP 429 rate limit or quota response.
- [ ] token.place HTTP 5xx provider/server response.
- [ ] token.place HTTP 200 with malformed completion payload.
- [ ] token.place timeout or aborted request.
- [ ] OpenAI selected with missing key.
- [ ] OpenAI authentication error when selected.
- [ ] OpenAI quota/rate-limit error when selected.

## 14) Rollback and mitigation plan

- [ ] Individual users can switch to OpenAI in `/settings` if they have their own OpenAI key.
- [ ] Deployment operators can temporarily point token.place URL/model to a known-good
      endpoint/model using `VITE_TOKEN_PLACE_URL` and `VITE_TOKEN_PLACE_CHAT_MODEL`.
- [ ] Operators can publish an outage/support note that explains token.place Chat is degraded and
      OpenAI remains optional for users with a key.
- [ ] Do not make OpenAI the default again without an explicit product decision.
- [ ] Do not add token.place credentials, API v2, streaming, or legacy relay endpoints as emergency
      workarounds.
